# Full System Audit: Scanner, Backtest, Live
## Date: 2026-03-25 | Auditor: Perplexity (fresh eyes, not CC)

---

## Executive Summary

**The system is overcomplicated to the point where the complexity itself is the primary source of bugs.** The strategy works. The exit logic works. But the infrastructure around it — three separate scanners, 259 env vars, 11,269 lines of core code, and layers of disabled features — creates a system where every fix introduces new inconsistencies and every backtest result is questionable.

**Recommendation: Targeted rebuild of the scanner pipeline. Keep the strategy logic (squeeze detector + exit logic in simulate.py). Gut and replace everything else.**

## Part 1: The Architecture Problem

### Three Scanners, Three Data Sources, Zero Parity

This is the root of every scanner bug CC has been chasing:

| Scanner | Used By | Data Source | Float Source | RVOL Computation | When It Runs |
|---------|---------|-------------|-------------|------------------|--------------|
| `scanner_sim.py` | Backtest (generates scanner_results/*.json) | Alpaca historical bars API | FMP → yfinance → EDGAR → AlphaVantage | PM-only volume ÷ Alpaca full-day ADV | Offline batch |
| `live_scanner.py` | Nobody — **NOT CONNECTED to bot.py** | Databento EQUS.MINI stream | Same chain + float_cache.json | Cumulative volume ÷ Databento ADV | Standalone process |
| `market_scanner.py` + `stock_filter.py` | Live bot (bot.py) | Alpaca snapshots API | Alpaca + yfinance | Alpaca daily_bar.volume ÷ 20-day avg | At startup + rescans |

**`live_scanner.py` is orphaned code.** Bot.py doesn't import or use it. The live bot uses `market_scanner.py` (Alpaca snapshots) → `stock_filter.py` for its stocks. The Databento scanner exists but feeds nothing.

**scanner_sim.py and the live bot use completely different data sources, completely different RVOL math, and completely different float lookups.** This is why scanner_sim finds 0 stocks on days the live bot finds 6. It's not one bug — it's a fundamental architecture mismatch.

### The RVOL Problem (Deeper Than CC Reported)

CC identified the RVOL bug: scanner_sim uses PM-only volume for rescan stocks. But the problem is worse:

1. **scanner_sim ADV** = Alpaca 20-day average daily volume (full-day, ~6.8M for FEED)
2. **live bot ADV** = Alpaca snapshot's `avg_volume` field (~115K implied for FEED)
3. These differ by **59x** — not because of a bug, but because they're different API endpoints returning different calculations

Even if you fix the PM-volume issue, the ADV denominator mismatch means RVOL will never match between backtest and live. A stock showing 19.2x RVOL in live might show 0.3x in the backtest, or vice versa.

### The Implication: All Backtests Are Suspect

Every megatest result, every YTD comparison, every A/B test — they all use scanner_results/*.json generated by scanner_sim.py. If scanner_sim systematically misses stocks that the live bot finds (or finds stocks the live bot misses), then:

- The +$130K megatest may be understated (missing rescan stocks)
- The +$25K YTD baseline may be understated
- The Ross Exit A/B comparison may be comparing different stock universes
- Strategy conclusions drawn from these backtests may be wrong

## Part 4: Recommendation — What to Rebuild

### The Scanner Pipeline: Gut and Replace

**Current state:** Three scanners, three data sources, irreconcilable differences.

**Proposed state:** ONE scanner, used by BOTH backtest and live, with the same data source.

#### Option A: Alpaca-Only Scanner (Simplest)

```
Single scanner module: unified_scanner.py
Data source: Alpaca API (you have premium — real-time snapshots)
Used by: bot.py (live) AND run_megatest.py (backtest via cached results)

Flow:
  1. Alpaca get_all_assets() → price filter ($2-$20) → ~3000 symbols
  2. Alpaca get_snapshots(symbols) → gap%, volume, prev_close
  3. Float lookup: FMP → yfinance → EDGAR → AlphaVantage (existing chain)
  4. Filter: gap ≥ 10%, RVOL ≥ 2x, float < 15M, PM vol ≥ 50K
  5. Rank top 5 by composite score
  6. Output: scanner_results/{date}.json (same format, same file)

For backtest: scanner_sim calls Alpaca historical bars API
  with the EXACT same filter logic, EXACT same RVOL math.
For live: bot.py calls the same filter functions at runtime.
```

**Pros:** Simplest. One data source, one code path. Alpaca premium gives you real-time snapshots. This is essentially what bot.py already does (market_scanner.py + stock_filter.py) — just unify it with scanner_sim.py.

**Cons:** Alpaca doesn't cover OTC. But we showed that OTC stocks are a small fraction of the opportunity, and you've deferred Polygon/IBKR for now.

#### Option B: Databento Historical + Alpaca Live (Current Intent, Done Right)

```
Backtest scanner: scanner_sim.py using Databento historical API
  (you have the subscription — use it for the data it's good at)
Live scanner: Alpaca snapshots (what bot.py already does)

BUT: Make them use THE SAME filter logic module.
  - shared_filters.py: gap_filter(), rvol_filter(), float_filter(), rank()
  - scanner_sim.py imports shared_filters
  - stock_filter.py imports shared_filters
  - RVOL computation: one function, parameterized by data source
```

**Pros:** Uses Databento's better tick data for backtesting. Keeps Alpaca for live (already working).

**Cons:** Still two data sources — RVOL will never be identical. But at least the filter logic is shared, so you'd only differ on the data inputs, not the business logic.

#### Option C: IBKR for Everything (Your Most Powerful Option)

You have Interactive Brokers with L2 capabilities. IBKR covers:
- ALL NMS stocks (NYSE, NASDAQ, AMEX)
- OTC/Pink sheets (with Penny Stocks permission)
- Real-time streaming data
- Historical data (bars and ticks)
- Built-in market scanner with programmable filters
- Pre-market data from 4 AM ET
- Paper trading for testing

```
Scanner: IBKR TWS API reqScannerSubscription()
  - Location: STK.US (all US stocks, including OTC)
  - Filters: price $2-$20, % change ≥ 10%, volume ≥ 50K
  - Returns: ranked list of gap-up small-caps
  - Real-time: updates every ~30 seconds automatically

Data feed: IBKR TWS API reqMktData() / reqHistoricalData()
  - Same source for live AND backtest
  - Same RVOL math because same underlying data

Execution: Keep Alpaca for NMS execution (already working)
  - Add IBKR for OTC execution (if you want OTC later)
```

**Pros:** Single data source for scanner, backtest, and live. OTC-ready. L2 data available for future strategies. IBKR's built-in scanner does exactly what scanner_sim.py tries to do — gap-up, high-volume, small-cap stocks — but it's maintained by a billion-dollar company.

**Cons:** Requires TWS/IB Gateway running on Mac Mini. Rate limits on historical data. More complex initial setup.

### My Recommendation: Option A Now, Option C Later

**Immediate (this week):** Unify scanner_sim.py and stock_filter.py into one shared filter module. Both use Alpaca data. One code path, one RVOL computation. This fixes the backtest ≠ live divergence with minimal work.

**Next month:** Evaluate IBKR as a replacement data source. You already have the subscription. The built-in scanner eliminates scanner_sim.py entirely. The historical data API gives you one source of truth for backtesting.

## Part 5: The Bigger Picture

CC has been doing good work — the strategy research, the exit analysis, the Ross video methodology extraction, the backtest framework. The problem isn't that the work is wrong. It's that **each improvement adds complexity, and the complexity compounds faster than the improvements.**

The WT scanner comparison proved the scanner selects good stocks (80% of P&L from 12% of stocks). The squeeze detector works (70% win rate, 21x profit factor). The exit logic works. But the scanner pipeline that feeds these systems is unreliable, and the codebase is so complex that bugs are introduced faster than they're fixed.

**The strategy is not the problem. The plumbing is the problem.**

Strip it down. One scanner, one data source, one code path. Let the strategy breathe on a clean foundation.
