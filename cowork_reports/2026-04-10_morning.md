## Root Cause

The Alpaca Python SDK uses `requests` underneath and **does not set a default HTTP timeout**. Combined with HTTP keep-alive connection pooling, this creates a latent bug:

1. Network blip causes a TCP socket in the Alpaca SDK's connection pool to go silently dead (no RST/FIN)
2. Next call to `state.alpaca.get_all_positions()` reuses the dead socket from the pool
3. Sends HTTPS request bytes onto a connection that will never get a response
4. `SSL_read_ex()` on the response → kernel `read()` syscall → blocks forever
5. The `try/except` in `periodic_position_sync()` doesn't catch this — **hangs aren't exceptions**, only timeouts raise

Process sample confirmed the exact stack at the moment of inspection (10+ hours after the freeze):

```
_PyEval_EvalFrameDefault
  → method_vectorcall
    → _ssl__SSLSocket_read_impl
      → SSL_read_ex
        → ssl3_read_internal
          → tls_default_read_n
            → BIO_read
              → sock_read
                → read (kernel)  ← blocked indefinitely
```

This was a latent bug the whole time. It only triggered today because today was the first day the bot was running through a real network connectivity blip. Earlier in the week:

- **Tue/Wed mornings:** Bot wasn't actually running (Gateway autostart broken)
- **Tue evening / Wed manual run:** No network blips, connection pool stayed clean
- **Today:** First morning blip → first hang

## Fix (commit cab9754, pushed to v2-ibkr-migration)

### Layer 1: Explicit timeouts on Alpaca calls

New helper `_alpaca_call(fn, *args, timeout=10, **kwargs)` runs the call in a `ThreadPoolExecutor` worker and uses `future.result(timeout=10)` to enforce a hard 10-second cap. If the call hangs, `TimeoutError` is raised and the existing `try/except` blocks handle it as a normal failure. The blocked worker thread is abandoned (we can't actually cancel a thread blocked on a kernel read), but the next call gets a fresh worker.

Applied to:
- `get_account_equity()` line 311
- `reconcile_positions_on_startup()` line 325
- `periodic_position_sync()` line 386 ← **the actual killer**
- `graceful_shutdown()` line 1986

### Layer 2: Main-thread watchdog

Daemon thread `_watchdog_loop` that:
- Wakes every 15s
- Checks `_last_heartbeat` timestamp (updated by main loop each iteration)
- If heartbeat is stale >120s, prints warning and calls `os._exit(1)`
- Cron's `check_bot.sh` will then restart the bot (the existing health check at 2:30 AM, plus we should consider running it more frequently)

`update_heartbeat()` is called inside the main `while True` loop right before `ib.sleep()`, which runs every 1 second in active mode. So the watchdog will fire after ~120s of total main-thread silence. Confirmed armed in startup log:

```
Watchdog: armed (kills bot if main thread frozen >120s)
```

### Why both layers?

Layer 1 fixes the specific bug we found. Layer 2 catches anything we missed — there are other Alpaca calls (`submit_order`, `cancel_order_by_id`, `get_order_by_id`) that aren't wrapped, plus `yfinance.Ticker().info` in `live_scanner.py` which has its own no-timeout problem. Without the watchdog, any of those could re-create the same incident.

## Other Alpaca calls NOT wrapped (intentional, for now)

These run in entry/exit code paths where a hang would already be catastrophic and the watchdog will catch it:
- `submit_order` (multiple sites)
- `cancel_order_by_id` (multiple sites)
- `get_order_by_id` (multiple sites)

If you want belt-and-suspenders, all of these can be wrapped with `_alpaca_call(...)` the same way. Let me know if that's worth doing or if the watchdog is enough.

## Known related bug: yfinance has no timeout

`live_scanner.py:226` — `yf.Ticker(symbol).info`

`yfinance` does not expose a timeout parameter. If FMP rate-limits and the float lookup falls through to yfinance, this can hang. The watchdog protects us, but it'd be cleaner to either skip yfinance entirely or wrap it in a thread/timeout the same way we did for Alpaca.

## Signal: SQFT cascade

This is the part worth your attention.

SQFT entered **3 times at the exact same level ($4.04)** and all 3 trades lost. The R-multiple decayed from -0.6R → -0.2R → -0.1R, which is the bot exiting earlier each time on tighter stops, but it's still entering the same failed level.

This is exactly the cascading-failed-resistance pattern that the resistance tracking work is meant to address (per CLAUDE.md known gaps: "Resistance tracking — bot enters same rejection zone repeatedly").

**X01 raised MAX_ATTEMPTS from 3 → 5**, which makes this worse on stocks that fade rather than cascade up. SQFT today is a textbook case: it broke $4.00 once at 07:01, failed, faded, came back to $4.04 at 09:21 and 11:10, failed both times.

Worth thinking about whether X01's "more re-entries" thesis (which scored +$37K in tuning) needs a counter-rule for "but only if the stock is making higher highs vs same level." The exhaustion filter was supposed to handle this but obviously didn't on SQFT.

Open questions for you:
1. Is there a tuning variant of X01 with `MAX_ATTEMPTS=3` we can backtest as a comparison? Was that explicitly tested or just a conclusion from the param sweep?
2. The R values dropped from 0.21 → 0.10 → 0.12 across the 3 entries — that's the squeeze detector finding tighter stops on later attempts. Should we be filtering out late re-entries with low R, since they're effectively gambling on a smaller move?
3. Resistance tracking proper would have blocked entries 2 and 3 here. Is that work scoped anywhere?

## Autostart Issue (still broken — day 3)

This is a separate issue from the hang, but worth re-flagging.

`daily_run_v3.sh` cron at 02:00 MT failed for the **third consecutive day** to bring up IB Gateway. Same symptom each time: IBC starts, Java process spawns, but port 4002 never opens within the 360-second timeout. Today's IBC log shows `"IBC: Starting Gateway"` as the last meaningful line before silence — never gets to login completion.

When CC manually kills the stale Java process and restarts IBC, Gateway comes up in **~10 seconds**. So the issue is that the cron's IBC instance is getting stuck on something the manual run doesn't hit. Possibilities:

1. **Stale Java from previous run not killed** — daily_run_v3.sh does `pkill -9 -f "java"` but maybe the IBC config dialog from the previous failed run is still around in some form
2. **Login dialog waiting for GUI input** — IBC's login automation might be failing silently if the screen isn't fully unlocked. The script does try to unlock the Mac via osascript keystroke, but logs `"WARN: keystroke failed — check Accessibility permissions"` consistently
3. **Cold cache vs warm cache** — manual restarts always happen after the cron tried first, so there might be some state from the first attempt that helps the second one succeed

This is starting to feel like a "fix it once, properly" rather than tweak-and-pray. Suggested directions:
- Add `pkill -9 -f "Trader Workstation Configuration"` and `pkill -9 -f "Pending Tasks"` to the cleanup before relaunching IBC
- Try `--on2fatimeout=restart` instead of `exit` in IBC args (currently `exit`)
- Move the IBC start to use a fully-headless mode if available (avoid all the dialog handling)
- OR just accept that cron is unreliable and write a launchd plist instead, which has better lifecycle management on macOS

CC will manually kick off the bot each morning until this is fixed.
