**Date:** April 7, 2026
**Author:** CC (Claude Code)

## Summary

The live bot receives significantly fewer trade ticks than what backtests use. This means volume-based thresholds (squeeze detector `vol_ratio`, volume floor, etc.) will compute differently live vs backtest. The gap is structural, not a bug.

---

## Three Data Sources in Play

| Source | Used By | API | Coverage |
|--------|---------|-----|----------|
| **Alpaca SIP** | Backtest tick cache (Jan-Mar) | `hist_client.get_stock_trades(feed="sip")` | Full consolidated tape — every trade print from all exchanges |
| **IBKR Historical Ticks** | Tick-level seeding (new) | `ib.reqHistoricalTicks()` | IBKR's historical trade data via HMDS |
| **IBKR RTVolume** | Live bot tick stream | `ib.reqMktData(genericTickList='233')` | Real-time aggregated feed — filtered/consolidated |

## Measured Gap: IBKR RTVolume vs IBKR Historical Ticks

**Test:** ADVB on April 7, 2026 — same stock, same 4-minute window, live vs historical.

| Metric | Live (RTVolume) | Historical (reqHistoricalTicks) | Ratio |
|--------|-----------------|--------------------------------|-------|
| Tick count | 1,028 | 4,074 | **4.0x** |
| Total volume | 68,493 | 239,905 | **3.5x** |

The live feed captured only **25% of the ticks** and **29% of the volume** that the historical API returns for the same period. This is IBKR's live feed aggregating/filtering rapid trade prints that the historical API preserves individually.

## Measured Gap: Alpaca SIP vs IBKR RTVolume

Not directly measured in the same window, but the pre-live tick caches (Jan-Mar) were generated by Alpaca SIP, which is the full consolidated tape. IBKR historical ticks are also comprehensive. So the gap is primarily:

**Backtest data (Alpaca SIP or IBKR Historical) >> Live data (IBKR RTVolume)**

## What This Means for the Strategy

### Squeeze Detector
- `vol_ratio` = current bar volume / average bar volume
- The squeeze arming threshold requires `vol_ratio >= WB_SQ_VOL_MULT` (default 3.0x)
- If the live feed only reports ~25-30% of true volume, but the AVERAGE is also built from the same live feed, the ratio MIGHT be self-consistent
- **Key question:** Does the RTVolume feed proportionally undercount in both the spike bar AND the average? If yes, the ratio is preserved. If the aggregation is inconsistent (e.g., more aggressive during high-volume spikes), the ratio is skewed.

### Box Strategy
- RSI and VWAP are price-based, not volume-based — less affected
- Volume confirmation (`BOX_MIN_BAR_VOL_RATIO=1.0`) could be affected
- But box uses a relative measure (current vs 20-bar average), so if the undercount is proportional, the ratio holds

## What We Need to Test

### Test 1: Ratio Consistency
For dates where we have BOTH live ticks and can pull historical ticks, compare the `vol_ratio` the squeeze detector would compute from each source. If the ratio is similar (within 20%), the detector will behave consistently even with fewer ticks.

### Test 2: Alpaca SIP vs IBKR Historical 
For the same stock-date, pull ticks from both Alpaca SIP and IBKR historical. If they match closely, our Jan-Mar backtest data is equivalent to what IBKR historical seeding provides. The only gap would be live RTVolume vs everything else.

### Test 3: Multi-Day Live Comparison
We now have April 6 and 7 with live tick caches. As more live days accumulate, we can compare backtest results (using live tick cache) with what the live bot actually did. Today (April 7) they matched perfectly — both 0 trades. April 6 had the seed bug which is now fixed.

---

## Possible Outcomes

### Scenario A: Ratios Are Self-Consistent (Best Case)
If RTVolume undercounts proportionally (both spikes and averages), then `vol_ratio` is preserved and the detector behaves the same live as in backtest. The absolute tick counts differ but the relative measures that drive decisions are equivalent. **Action: No changes needed.**

### Scenario B: Ratios Are Skewed (Worst Case)
If RTVolume compresses high-volume spikes more than low-volume bars, then `vol_ratio` is systematically lower live than backtest. **Action: Calibrate `WB_SQ_VOL_MULT` threshold lower for live trading (e.g., 2.0x instead of 3.0x), or add a live-mode scaling factor.**

### Scenario C: Need Different Data Source
If the gap is too large to calibrate, switch the live bot's tick feed to something richer. Options:
- Alpaca real-time SIP feed (if available for IBKR-first stocks)
- Databento real-time feed
- IBKR's `reqTickByTickData('AllLast')` instead of RTVolume (tick-by-tick API, more granular but higher overhead)
