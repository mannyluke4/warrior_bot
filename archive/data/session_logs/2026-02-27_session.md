## Critical Bug #1: IEX vs SIP WebSocket Feed

### The Discovery
While investigating why BATL's backtest showed +$4,450 but the live bot skipped it entirely, we compared Alpaca REST bar data against live bot bar data for the same 08:14 minute:

| Source | Close | High | Volume |
|--------|-------|------|--------|
| Alpaca REST (SIP) | $4.84 | $4.84 | 56,434 |
| Live Bot WebSocket (IEX) | $4.97 | $4.97 | 103 |
| Backtest (SIP ticks) | $4.84 | $4.84 | ~56K |

**Volume discrepancy: 545x** — the live bot saw 103 shares where 56,434 actually traded.

### Root Cause
```python
# data_feed.py — AlpacaFeed.__init__
from alpaca.data.live import StockDataStream
self._stream = StockDataStream(api_key, api_secret)
# StockDataStream defaults to feed=DataFeed.IEX (free tier, single exchange)
```

The `StockDataStream` constructor defaults to `feed=DataFeed.IEX`, which only receives trades from the IEX exchange — a tiny fraction of total market volume. The backtester uses `feed="sip"` for REST bar/trade fetches, which is the full consolidated tape across all exchanges.

This means:
- **Every live trade the bot has ever taken** was based on IEX-only data (partial bars, wrong prices)
- **Every backtest** used SIP data (complete bars, accurate prices)
- **Backtests and live were never seeing the same data** — tuning backtests had a fundamental gap
- The user pays for Algo Trader Plus (includes SIP) but the bot wasn't using it

### The Fix (Committed: ce5d427)
```python
# data_feed.py — AlpacaFeed.__init__ (one line change)
```

## Critical Bug #2: Orphaned Orders (entry_give_up doesn't cancel)

### The Discovery
After the session, user noticed an open FIGS position in Alpaca that the bot had no awareness of. The bot's logs showed zero fills — it "gave up" after max reprice attempts but never cancelled the Alpaca order.

### Root Cause (Two bugs combined)

**Bug A: `entry_give_up` doesn't cancel the Alpaca order**

In `trade_manager.py`, both give_up paths (`max_attempts` and `at_max_chase_cap`) called `self.pending.pop(symbol, None)` to remove the order from bot memory, but **never called `cancel_order_by_id()`**. The order remained live in Alpaca and eventually filled — just 3 seconds after the bot "gave up."

**Bug B: Reconcile thread doesn't check filtered-out symbols**

The reconcile universe is `filtered_watchlist ∪ open ∪ pending ∪ pending_exits`. FIGS was filtered out at startup (float 152M > 50M max), so it wasn't in `filtered_watchlist`. After `entry_give_up` popped it from `pending`, FIGS wasn't in ANY set — the reconcile safety net never checked it.

### Contributing Factor: WebSocket Crash
A websocket `keepalive ping timeout` error occurred at ~09:34 ET, crashing the `watchlist_thread`. This meant the bot couldn't receive fill notifications even if the order status changed. The pending_heartbeat thread was still running but had already forgotten about FIGS.

### The Fixes (trade_manager.py)

**Fix A: Cancel order on give_up**
```python
# Both max_attempts and at_max_chase_cap paths now cancel before returning:
self.pending.pop(symbol, None)
try:
    self.client.cancel_order_by_id(order_id)
    log_event("entry_give_up_cancel_sent", symbol, order_id=order_id)
except Exception as e:
    log_event("entry_give_up_cancel_failed", symbol, order_id=order_id, error=str(e))
```

**Fix B: Sweep ALL Alpaca positions every ~30s**
```python
def _reconcile_all_positions(self):
    """Query ALL Alpaca positions and adopt any the bot doesn't know about."""
    positions = self.client.get_all_positions()
    for pos in positions:
        if sym not in self.open and sym not in self.pending:
            # ORPHAN DETECTED — adopt it
            self._ensure_open_trade_from_alpaca(sym, alp_qty, alp_avg)
```
Called every 10th reconcile cycle (~30 seconds). Logs `reconcile_orphan_detected` and prints a warning.

### Impact
- Prevents future orphaned positions from entry give_ups
- Safety net catches ANY position the bot doesn't know about (regardless of cause)
- WebSocket crashes, API errors, or any other failure mode that could create orphans will be detected within 30 seconds

## Feature Developed: Volatility Floor (Vol Floor)

Implemented `_vol_floor_stop()` in micro_pullback.py. Two mechanisms:
1. ATR-based: min R = mult * avg_bar_range
2. Price-based: min R = pct% of entry price

**ANNA with 5% vol floor:** Loss cut from -$1,088 to -$537 (backtest). Wider stop survived dip, bearish engulfing exit instead of hard stop.

**Regression result (10 stocks, 5% floor):** -$1,704 vs baseline. Helps shakeout stocks (PAVM +$4,624) but hurts spike-capture winners (GWAV -$4,034). The 5% blanket is too blunt — needs smarter activation criteria.

**Status:** Code implemented, env-gated OFF by default (`WB_VOL_FLOOR_ENABLED=0`). Needs web Claude review for parameter tuning. See [VOL_FLOOR_PROPOSAL.md](VOL_FLOOR_PROPOSAL.md) for full analysis with 7 tuning options.

## Key Takeaways

1. **The SIP fix is the single most impactful change since the bot was built.** Every indicator, every bar, every entry/exit calculation was based on partial exchange data. This explains many historical live vs backtest discrepancies.

2. **Re-entry is the #1 source of losses.** Three stocks with double losses from re-entry = -$3,384, or 60% of total session losses. The bot doesn't learn from failure on a symbol.

3. **The orphan bug could have been catastrophic.** FIGS happened to go up. If it had crashed, the bot would have had an unmonitored, unmanaged loss with no stop protection.

4. **Topping wicky is killing winners.** ARLO captured 7% of available profit. This is a recurring pattern from the trade profile study.

5. **Low-score, no-tag entries are consistently losers.** MRM (5.5), LBGJ (5.5/4.0) — combined -$2,106. These entries have no structural basis.

6. **Buying power cascade is real.** Early losses on weak setups depleted capital, blocking the best setup (FIGS, score 12.5). The bot should prioritize high-conviction entries.

7. **Tight R + massive position size remains the core risk.** ANNA ($0.09 R, 11K shares), ONMD T2 ($0.07 R, 14K shares) — the vol floor addresses this but needs activation criteria tuning.
